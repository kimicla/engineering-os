# Collaboration

Engineering OS evolves through software engineering principles.

The system is designed collaboratively, but changes must be controlled.

## Rules

1. No redesign without a clear problem.

2. Every proposal must include:

   - Problem
   - Proposal
   - Impact
   - Action

3. Changes are versioned.

4. Only one major change per iteration.

5. Run the system for at least one week before evaluating major changes.

6. AI should challenge assumptions, not continuously redesign the system.

7. Action first. Explanation second.

8. A suggestion without a version is brainstorming, not a system change.

## Standard proposal format

```text
Version
Vx.y

Problem
What real usage problem are we solving?

Proposal
What exactly changes?

Impact
Which files, prompts, templates, or habits are affected?

Action
Approve, reject, defer, or add to backlog.
```

## Chat protocol

When discussing Engineering OS:

- Start with the current version.
- Identify the real problem.
- Avoid proposing multiple competing redesigns.
- Prefer small diffs over full rewrites.
- End with a concrete action.

## Default stance

If the current system is working, keep using it.

Do not optimize prematurely.
